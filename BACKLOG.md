Заложить структуру монорепо (/api, /backend, /frontend), Node-тулчейн с фиксированными версиями @typespec/*, команду генерации и политику версионирования — фундамент для последующих эпиков (кодогенерация бэкенда M2, фронтенд M3).

### 1.4 Описать в .tsp чистовые модели и операции для всех текущих эндпоинтов + исправить именование

Текущая поверхность API (из `internal/handler/post_handler.go`):

### 1.5 Доказать parity контракта: oasdiff сгенерированного OpenAPI против текущего swaggo-спека + коммит артефакта

**Веха:** M1 — Contract Freeze (API in TypeSpec) · **Приоритет:** P0 · **Оценка:** M · **Метки:** type:chore, area:contract, area:ci, priority:P0

**Зависит от:** Описать в .tsp чистовые модели и операции для всех текущих эндпоинтов + исправить именование

**Контекст**

Цель эпика — TypeSpec должен воспроизводить ТЕКУЩИЙ API точно. Нужно объективно доказать отсутствие ломающих различий между новым сгенерированным контрактом и текущим свагго-спеком (`docs/swagger.json` / `docs/swagger.yaml`, Swagger 2.0), а затем закоммитить сгенерированный OpenAPI как артефакт-источник правды. oasdiff позже станет CI-гейтом на breaking changes (это часть эпика гардрейлов, но первичную проверку parity делаем здесь вручную/скриптом). Важный нюанс parity: намеренные исправления именования (`responce`→`response`, `Platfroms`→`Platforms`, `Sheduled_for`→`ScheduledFor`) и добавленные `/auth/*` эндпоинты — это ОСОЗНАННЫЕ расхождения. ### 3.6 Google OAuth логин и обработка сессии/токенов на клиенте

**Веха:** M3 — Frontend MVP · **Приоритет:** P0 · **Оценка:** L · **Метки:** type:feature, area:frontend, priority:P0

**Зависит от:** App shell, роутинг, границы загрузки/ошибок и конфиг базового URL API через env; Настройка Redux Toolkit store для клиентского/UI-состояния; Генерация типизированного клиента и хуков TanStack Query из OpenAPI (@hey-api/openapi-ts)

**Контекст**

Аутентификация на бэкенде уже реализована через Google OAuth (goth/gothic) и JWT — см. `internal/handler/post_handler.go` и `internal/auth/`. Реальный контракт потока:

- `GET /auth/google` — начинает OAuth (`beginAuthFunction`, редирект на Google).
- `GET /auth/google/callback` (`getAuthCallbackFunction`) — по успеху возвращает JSON `{ "access_token": "<JWT>" }` и ставит httpOnly Secure cookie `refresh_token` (path `/auth/refresh`, срок 30 дней).
- `POST /auth/refresh` (`refreshTokensHandler`) — по cookie `refresh_token` выдаёт новый `{ "access_token": ... }` и ротирует refresh-cookie.
- Защищённые эндпоинты требуют заголовок `Authorization: Bearer <accessToken>` (`AuthMiddleware`), access-токен живёт 15 минут (`auth.GenerateTokens` в `internal/auth/jwt.go`), при истечении — 401.

Задача — реализовать этот поток на клиенте: логин, хранение access-токена, автоматический refresh, выход. Refresh-токен — httpOnly cookie, из JS недоступен, поэтому фронт хранит только access-токен (в памяти) и полагается на cookie для обновления.

**Что сделать**

- Экран `/login` (Mantine): кнопка "Войти через Google", которая уводит на `GET {VITE_API_BASE_URL}/auth/google` (полный редирект браузера, не fetch).
- Обработать возврат после callback: бэкенд отдаёт `access_token` в JSON — согласовать с бэкендом целевой UX (редирект на страницу фронта с передачей токена ИЛИ отдельный вызов; если поведение бэкенда нужно поменять — завести follow-up на area:backend, т.к. текущий callback просто отвечает JSON). Зафиксировать выбранный вариант в PR.
- Хранить access-токен в памяти (Redux `authSlice` или React context) — НЕ в `localStorage` (XSS-риск); refresh-токен остаётся в httpOnly cookie и клиентом не читается.
- Подключить access-токен к сгенерированному клиенту: interceptor/`setConfig` (точка расширения из задачи про генерацию клиента) добавляет `Authorization: Bearer <accessToken>` ко всем защищённым запросам.
- Реализовать авто-refresh: при `401` от API один раз дёрнуть `POST /auth/refresh` (с `credentials: 'include'`, чтобы ушла cookie), при успехе — обновить access-токен и повторить исходный запрос; при неудаче — очистить сессию и редирект на `/login`.
- `ProtectedRoute`/guard: неаутентифицированного пользователя редиректить на `/login`; после логина возвращать на исходный путь.
- Кнопка "Выход": очистить access-токен из памяти, инвалидировать кэш TanStack Query (`queryClient.clear()`), редирект на `/login` (при необходимости — эндпоинт разлогина завести follow-up на backend).
- Убедиться, что dev-прокси (`vite.config.ts`, задача про app shell) корректно передаёт cookie `refresh_token` на бэкенд при `POST /auth/refresh`.

**Критерии приёмки**

- [ ] На `/login` кнопка "Войти через Google" инициирует `GET /auth/google` и проходит реальный OAuth-редирект.
- [ ] После успешного логина access-токен сохраняется в памяти клиента (не в localStorage) и подставляется в `Authorization: Bearer` для защищённых запросов.
- [ ] При `401` выполняется однократный `POST /auth/refresh` (с cookie), успешный refresh прозрачно повторяет исходный запрос; при провале — редирект на `/login`.
- [ ] `ProtectedRoute` защищает `/posts`, `/posts/new`, `/posts/:id/edit`; неавторизованный уходит на `/login`.
- [ ] Кнопка выхода чистит токен и кэш TanStack Query и уводит на `/login`.
- [ ] Refresh-токен не читается из JS; хранится только как httpOnly cookie.
- [ ] Любые требуемые изменения контракта бэкенда (поведение callback/эндпоинт logout) оформлены отдельными follow-up с меткой area:backend.

**Definition of Done**

### 4.5 Убрать хак «+3 часа»: хранить и сравнивать время в UTC, конвертировать на границе API

**Веха:** M4 — Data Layer (sqlc + pgx v5) · **Приоритет:** P1 · **Оценка:** M · **Метки:** type:tech-debt, area:db, priority:P1

**Зависит от:** Переключить internal/repository на sqlc-методы за существующим интерфейсом (без изменения API)

**Контекст**

### 5.5 Детект breaking changes контракта на PR через oasdiff GitHub Action

### 6.1 Graceful shutdown по SIGTERM: сквозная отмена контекста и закрытие ресурсов

**Веха:** M0 — Foundations & Decisions · **Приоритет:** P0 · **Оценка:** L · **Метки:** type:tech-debt, area:backend, priority:P0

**Контекст**

Объединяет #27. Сейчас завершение работы сломано на уровне контекста. В `main.go` создаётся `ctx := context.Background()` без `cancel`, и этот `ctx` прокидывается в `app.NewApp` и используется планировщиком (`a.Scheduler.Start(a.Ctx)`) и Kafka-консьюмером (`reader.ReadMessage(ctx)`).

Критический баг в `internal/app/app.go`: в `NewApp` вызывается `_, cancel := context.WithCancel(context.Background())` — производный контекст отбрасывается (`_`), а в `App.Cancel` кладётся `cancel` от него. При этом `App.Ctx` = переданный `ctx` (по факту `context.Background()`). Поэтому `a.Cancel()` в `WaitForShutdown` отменяет контекст, который никто не слушает: `a.Ctx.Done()` в планировщике и воркере не сработает никогда.

HTTP-сервер запускается через `r.Run(":8080")` в горутине и никогда не останавливается gracefully. `WaitForShutdown` вызывает `a.Wg.Wait()`, который может зависнуть (см. отдельную задачу про воркер). Плюс перед запуском Kafka-консьюмера стоит `time.Sleep(30 * time.Second)`.

**Что сделать**

- В `main.go` создать корневой контекст с отменой по сигналам: `ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)` и передать его в `NewApp`; убрать ручной канал сигналов.
- Исправить `NewApp` в `internal/app/app.go`: строить производный контекст от переданного `ctx` (`ctx, cancel := context.WithCancel(ctx)`), сохранять в `App.Ctx` именно отменяемый контекст и `App.Cancel = cancel`. Убедиться, что `a.Ctx` и `a.Cancel` связаны.
- Заменить `r.Run(":8080")` на `http.Server{Addr: ":8080", Handler: r}` с `srv.Shutdown(shutdownCtx)` при завершении; таймаут shutdown ~2s через `context.WithTimeout`.
- Реализовать корректную последовательность остановки: отмена контекста -> остановка планировщика и Kafka-консьюмера -> `srv.Shutdown` -> `a.Wg.Wait()` (с таймаутом) -> закрытие пулов БД (`dbpoolmaster.Close()`, `dbpoolslave.Close()`) и `kafkaProducer.Close()`. Уложиться в ~2s.
- Убрать `time.Sleep(30 * time.Second)` перед запуском Kafka-консьюмера (задержку заменить корректным ожиданием готовности брокера/ретраями подключения при необходимости).
- Экспортировать/добавить метод для остановки Kafka-продюсера (`kafka.Producer.Close()` уже есть) и вызывать его при shutdown.

**Критерии приёмки**

- [ ] `App.Ctx` — производный от корневого контекста; `a.Cancel()` реально приводит к срабатыванию `a.Ctx.Done()` в планировщике и воркере.
- [ ] Отправка SIGTERM (или Ctrl+C) приводит к остановке планировщика, Kafka-консьюмера, graceful-остановке HTTP-сервера и закрытию пулов master/slave в пределах ~2 секунд без зависаний.
- [ ] `time.Sleep(30 * time.Second)` перед Kafka-консьюмером удалён.
- [ ] `a.Wg.Wait()` не зависает при завершении (проверяется совместно с задачей про воркер).
- [ ] Нет утечки: `srv.Shutdown` и `Close()` пулов/продюсера вызываются ровно один раз.

**Definition of Done**

Код собирается (`go build ./...`), проходит `go vet ./...` и существующие тесты. Ручная проверка: запуск через docker-compose, `docker compose stop hexlet-project` (SIGTERM) — логи показывают упорядоченное завершение всех компонентов за ~2s. Изменения в `main.go` и `internal/app/app.go` отражены в описании PR со ссылкой на #27.

### 6.2 Исправить одноразовый background-воркер и утечку горутины

**Веха:** M0 — Foundations & Decisions · **Приоритет:** P0 · **Оценка:** M · **Метки:** type:tech-debt, area:backend, priority:P0

**Зависит от:** Graceful shutdown по SIGTERM: сквозная отмена контекста и закрытие ресурсов

**Контекст**

Объединяет #27. В `internal/app/app.go` метод `backgroundWorker(msg)` устроен так, что обрабатывает сообщение ровно один раз и с искусственной задержкой:

```
ticker := time.NewTicker(10 * time.Second)
select {
case <-a.Ctx.Done():
    return
case <-ticker.C:
    a.proccesProcessing(msg)
}
```

Проблемы: (1) `select` срабатывает один раз, обработка каждого сообщения искусственно откладывается на 10 секунд без причины; (2) горутина учитывается в `a.Wg`, но пока `a.Ctx` не отменялся (см. баг контекста в связанной задаче), при остановке `a.Wg.Wait()` может зависать до тех пор, пока не отработают все тикеры; (3) `StartBackgroundWorker` запускает по горутине на каждое сообщение без ограничения параллелизма.

**Что сделать**

- Убрать `time.NewTicker(10 * time.Second)`: обрабатывать сообщение сразу, а `a.Ctx.Done()` проверять для быстрого выхода (`select { case <-a.Ctx.Done(): return; default: }` или передавать `a.Ctx` в `proccesProcessing`).
- Прокинуть `a.Ctx` во все вызовы репозитория внутри `proccesProcessing`, чтобы при отмене они прерывались.
- Убедиться, что `a.Wg.Add(1)` в `StartBackgroundWorker` и `defer a.Wg.Done()` в `backgroundWorker` парные и `Wg.Wait()` завершается сразу после отмены контекста.
- Рассмотреть ограничение числа параллельных воркеров (буферизованный семафор-канал) — при большом потоке сообщений не плодить неограниченное число горутин. Если решено не вводить сейчас — зафиксировать это решение комментарием и в описании PR.

**Критерии приёмки**

- [ ] Сообщение обрабатывается без искусственной задержки 10s.
- [ ] При отмене `a.Ctx` воркеры завершаются, `a.Wg.Wait()` возвращается без зависания.
- [ ] Нет утечки горутин: после shutdown нет висящих воркеров (проверка `go test -race` или ручная через `runtime.NumGoroutine`).
- [ ] Решение по ограничению параллелизма принято и зафиксировано (введено или явно отложено).

**Definition of Done**

`go build ./...`, `go vet ./...` и `go test ./...` зелёные. В PR указана ссылка на #27 и связанную задачу про graceful shutdown, от которой зависит корректность отмены контекста.

### 6.6 Безопасные type assertions для JWT-клеймов и currentUserID (401 вместо паники)

**Веха:** M0 — Foundations & Decisions · **Приоритет:** P0 · **Оценка:** S · **Метки:** type:tech-debt, area:backend, priority:P0

- В `CreatePlatform` перестать игнорировать флаг наличия: использовать `val, exists := rw.Get("currentUserID")` с проверкой `exists`.
- Рассмотреть вынос получения `currentUserID` в helper-функцию, чтобы не дублировать проверку в 9 местах.

**Критерии приёмки**

- [ ] Нигде в `post_handler.go` нет прямого `.(string)` без `, ok`.
- [ ] Отсутствие/некорректный `sub` в refresh-токене возвращает 401, а не панику/500.
- [ ] Отсутствие `currentUserID` в контексте возвращает 401 во всех хендлерах, включая `CreatePlatform`.

### 7.5 Завести набор ADR: 0001 Design First, 0002 oapi-codegen, 0003 TanStack Query, 0004 отложить sqlc

**Веха:** M0 — Foundations & Decisions · **Приоритет:** P1 · **Оценка:** M · **Метки:** type:docs, area:docs, priority:P1

  - `0001-design-first.md` — единый источник правды TypeSpec → OpenAPI 3.1; уход от текущего code-first swaggo/Swagger 2.0 (`docs/swagger.json` = `"swagger":"2.0"`). Последствия: контракт первичен, генерация FE/BE из спеки.

  - `0003-tanstack-query-over-rtk-query.md` — серверное состояние через TanStack Query (клиент из `@hey-api/openapi-ts` + TanStack Query plugin), Redux Toolkit — ТОЛЬКО для клиентского/UI-состояния. RTK Query отклонён (смешивание с TanStack для server state — антипаттерн).
  - `0004-defer-sqlc.md` — переход на sqlc + pgx v4→v5 откладывается в отдельный поздний эпик (M4).
