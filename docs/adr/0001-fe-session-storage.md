# ADR 0001. Хранение сессии на фронтенде: httpOnly-куки, не localStorage

- Статус: Принято
- Дата: 2026-07-09
- Область: фронтенд (SPA). Серверная реализация (выдача кук, CSRF) — вне этого ADR, трекается в #240 (BE) и #82.
- Связанные тикеты: #217 (EPIC «Безопасность»), #110, #112, #187, #188, #214, #216, #166, #134.

## Контекст

Бэкенд выдаёт access-JWT и refresh-токен. Фронт пока работает на моках: реальных
auth-фич (#110/#112/#187/#188) ещё нет, поэтому момент выбора «где SPA хранит
access-токен» — впереди. Это foundational-решение: если написать auth-код с токеном
в `localStorage`, а потом менять стратегию, придётся переделывать весь клиент. Значит
решение фиксируем ДО написания auth-кода.

Модель угроз:

- Любой XSS (риски: RichText-редактор #195, HTML-превью #183) может прочитать
  `localStorage`/`sessionStorage` из JS и увести bearer-токен → угон сессии.
- Продукт публикует в чужие соцсети от имени пользователя (воркер #134) и содержит
  биллинг (#166). Угон сессии = публикации от чужого имени и доступ к платёжным данным.
- Токен/refresh/PII в Redux-сторе (легко дампится через Redux DevTools) или в
  `console.*`/логах ошибок — та же утечка, даже без XSS.

## Решение

1. **Access и refresh хранятся только в куках `HttpOnly; Secure; SameSite`.**
   Фронт токен в JS не видит и не хранит. httpOnly-кука недоступна из JavaScript,
   поэтому XSS не может её прочитать.

2. **Авторизация запросов — через `credentials: 'include'`.** Кука уходит с каждым
   запросом автоматически; отдельная передача токена из JS не нужна.

3. **Токены, refresh и PII (email, платёжные данные) НЕ кладём в Redux и в persist.**
   В сторе — только UI-состояние и, при необходимости, несекретные поля для интерфейса
   (`userId`, имя, роль). Никакой персист сессии на клиенте.

4. **Факт «пользователь залогинен» определяем по ответу `GET /me` (#112)** через
   TanStack Query, а не по наличию токена в клиентском хранилище.

5. **Логи не содержат секретов.** Ни `console.*`, ни обработчик ошибок (Error Boundary
   из #214) не печатают заголовки, куки или тела auth-ответов.

6. **`VITE_*` — только публичные значения.** В бандл не попадают секреты
   (`shared/config/env.ts` держит лишь `apiUrl`). Пересекается с #216.

## Соответствие текущего кода

На момент ADR фронт УЖЕ соответствует решению — переписывать ничего не нужно, решение
лишь фиксирует статус-кво и запрещает регресс:

- `src/shared/api/client.ts` — `fetch(..., { credentials: 'include' })`, токены не
  трогает и не хранит.
- Redux (`src/app/store`, слайсы `entities/session`) — только UI-состояние. Мок-сессия
  `session` хранит один булев флаг `isAuthenticated`, без токенов и PII.
- `grep -rn "localStorage\|sessionStorage" frontend/src` — пусто.
- `console.*` в `frontend/src` — пусто.
- `shared/config/env.ts` — единственное значение `apiUrl`, секретов нет.

## Guardrails

- ESLint: `no-restricted-globals` и `no-restricted-properties` на `localStorage`/
  `sessionStorage` (см. `frontend/eslint.config.js`). Мягкий барьер — не даёт случайно
  завести хранение сессии в web storage; на текущий код не влияет (использований нет).
- Ревью: PR, добавляющий токен/refresh/PII в Redux, persist или логи, не проходит ревью.

## Импликации для бэкенда (вне scope FE, трекается отдельно)

Раз кука ходит автоматически, нужна защита от CSRF для мутаций (POST/PUT/DELETE):
`SameSite=Lax/Strict` + проверка `Origin`/`Referer`, либо double-submit CSRF-токен.
`SameSite=Lax` не защищает top-level GET-навигацию — все изменяющие операции должны быть
не-GET (совпадает с планом ухода от тела в GET, см. комментарий в `client.ts` и #45).
Для OAuth-редиректов (#188) при необходимости — `SameSite=Lax` вместо `Strict`.

## Последствия

- Плюсы: XSS не крадёт сессию; клиент проще (не управляет токенами); отзыв/ротация
  refresh — на стороне сервера.
- Минусы/обязательства: обязательна CSRF-защита на бэкенде; `Secure`-куки требуют HTTPS;
  локальная разработка ходит через dev-прокси (`vite.config.ts`), same-origin.

## Альтернативы (отклонены)

- **Токен в `localStorage`/`sessionStorage`** — читается любым XSS, отклонено.
- **Токен в памяти JS (in-memory)** — переживает XSS не лучше и теряется при перезагрузке,
  усложняет клиент; для нашего профиля угроз httpOnly-кука строго предпочтительнее.
